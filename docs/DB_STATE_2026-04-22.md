# Database State — April 22, 2026 (end of day)

## Current state (stable)

**All services write to `/root/Mining-Gaurdian/guardian.db`** (the legacy monolithic SQLite file, ~6.6 GB).

Split DBs under `/root/Mining-Gaurdian/databases/` and the PostgreSQL instance `mining_guardian` are retained as **cold snapshots** — not written to, not read from by any service. They represent point-in-time copies taken during April 21-22 migration attempts.

## What happened this session

1. **Started the day:** mining-guardian.service was crash-looping since ~15:40 yesterday with `NameError: json is not defined` in `core/database.py:save_notifications`.
2. **Fixed the json import.** mining-guardian then got further into the scan loop and hit a second crash: `sqlite3.OperationalError: no such table: miner_restarts` in `count_outcome_failures`.
3. **Root cause of second crash:** `core/database.py` was extracted from the monolith on April 21 and several methods issue cross-table queries (e.g. opens connection to `audit.db` but SELECTs from `miner_restarts` which lives in `operational.db`). SQLite cannot span these without ATTACH. The split DB routing is structurally incompatible with the extracted database layer until every method is audited.
4. **Decision:** pause the split-DB migration. Pointed `_connect()` in `core/database.py` back at the legacy `guardian.db`. All 38 call sites resolve to one DB, all cross-table joins work, no other file changes needed.
5. **Verified:** scan 1678 wrote successfully at 16:37:45 to legacy guardian.db. Service is stable.

## Files modified

- `core/database.py` — added `import json`; rewrote `_connect()` to return a contextmanager yielding a connection to the legacy monolithic db. `table_name` argument kept for API compatibility but ignored. Router imports remain but are unused.
- Backups: `core/database.py.prefix-json` (before json fix), `core/database.py.before-revert` (before _connect revert).

## What's unchanged

- `core/database_router.py` — unchanged, now unused
- `core/db_compat.py` — unchanged, still points at legacy guardian.db
- `core/db_helper.py` — unchanged, still points at legacy guardian.db
- The 7 services using `db_compat`/`db_helper` — unchanged, still writing to legacy guardian.db as they have been
- Split SQLite DBs (`databases/*.db`) — frozen, not being written to
- PostgreSQL `mining_guardian` DB — frozen, not being written to

## Recovery

To revert to the split-DB router behaviour (not recommended until the cross-table queries in database.py are fixed):
```
cp core/database.py.before-revert core/database.py
systemctl restart mining-guardian
```

## Open issues inherited from this work

1. **core/database.py has cross-table-join bugs** that assumed one database. These surface as `no such table: X` errors when the router is active. Every method that does a SELECT/JOIN needs review before the split can be revived.
2. **The split SQLite migration and PostgreSQL migration are both half-finished.** Neither is production-ready. Postgres is additionally missing ~30 secondary indexes that failed to create when the migration script errored on `ams_notifications(miner_id)` (wrong column name in the index script).
3. **A real database strategy decision is still pending.** Options: stay on single SQLite forever, fix the split-DB cross-joins, do a clean PostgreSQL migration with proper indexes (preferably as part of the Mac Mini cutover in May). No rush — single SQLite at 6.6GB is fine for a 49-miner POC.

## What tomorrow-you should NOT do

- Do not enable the router until the `core/database.py` cross-table-join bugs are fixed
- Do not `cp config_template.json config.json` on VPS (causes credential loss — this happened twice before)
- Do not delete `databases/*.db` or drop the PG database — those are your rollback points if anything weird shows up in guardian.db
