## Failure modes spotted this session

- **Carve-outs that quietly become stale.** The original P-023 guard
  carved out `scripts/backup_db.sh` and `scripts/backup_mining_guardian.sh`
  from the VPS-IP allowlist on the assumption they were operator-only
  and never shipped. They WERE shipping (via `--include 'scripts/***'`).
  P-024 fixed that by narrowing the rsync, but the original carve-out
  comment still has to be read carefully. Lesson: when adding a
  carve-out to a CI guard, document AND verify the underlying
  assumption.
- **Rsync filter pattern matching.** `--include 'branding/***'` looks
  like it only matches the top-level `branding/` directory but
  matches anywhere — including `scripts/branding/`. Discovered
  during P-024 static replay. Fixed with anchored `/branding/***`.
  Lesson: any time an rsync filter pattern is added without a leading
  `/`, ask "what else can this match?"
- **Pre-existing test asserting the OLD include literal.** P-025's
  installer-resources test §7 had `--include 'scripts/***'` baked in
  as a string assertion. Without updating it in the same commit as
  the build_pkg.sh change, P-024 would have produced a regression
  there. Caught during the focused-test run before push. Lesson: a
  filter narrowing is a contract change; every test that pinned the
  old contract has to migrate too.
- **macOS `/tmp` → `/private/tmp` realpath quirk.** P-022's discovery
  sink test passed on Linux and failed on Bobby's macOS validation
  gate because `Path("/tmp/x").resolve()` follows the symlink to
  `/private/tmp/x`. Fixed with `os.path.realpath()` on both sides of
  the equality. Lesson: any test that compares paths across `/tmp`
  must canonicalize; this is a perennial trap.
