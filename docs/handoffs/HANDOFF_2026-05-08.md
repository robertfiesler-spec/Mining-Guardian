### What the live install of `2b41764` nevertheless exposed (B-45 / P-036)

The very same scan that proved P-034 / P-035 worked also exposed a
new failure mode that the P-035 test suite did not catch because no
P-035 test exercised a P-029-shaped layout (canonical file under
`knowledge/knowledge.json` + a compat symlink at `knowledge.json`
pointing at it):

- At 15:43 the scanner's first atomic write through the compat
  symlink replaced the **symlink** at
  `${MG_INSTALL_ROOT}/knowledge.json` with a root-owned **regular
  file of 948 bytes**.
- The canonical file at `${MG_INSTALL_ROOT}/knowledge/knowledge.json`
  was **untouched** (3,739,968 bytes; 96 / 133 / 176 as listed in the
  YAML above).
- Root cause: `core.file_lock.locked_knowledge_update` and
  `atomic_write_json` create a temp file in the parent directory of
  the path they were given and call `os.replace(tmp, path)`. When
  `path` is a symlink, POSIX `rename` overwrites the **symlink**
  with the regular temp file rather than updating the target. Every
  active writer computes its target as `_ROOT / "knowledge.json"`,
  which in the install layout is the compat symlink — so the first
  writer to fire after install broke the compat path immediately.
- Reads through the canonical `knowledge/knowledge.json` path kept
  working; reads through the symlink path silently saw a stale
  948-byte file. A divergence that would compound over time. Vision
  Anchor 1 risk.

### P-036 fix landed on `main` after the mitigation

PR #169 (`mg/p036-knowledge-symlink-preserve-canonical-resolver`,
squash `906fa4a`, merge `fab6694`) was opened immediately after the
mitigation, reviewed, and merged. It changes ONLY `core/file_lock.py`
so that `locked_knowledge_update` and `atomic_write_json` resolve a
symlink target one level before computing the temp directory and the
rename destination — temp file lands in the **target's** parent and
`os.replace` lands on the canonical file, leaving the symlink itself
untouched. The lock file is also keyed off the resolved target so two
writers — one given the symlink, one given the canonical path —
serialize against the same flock. Non-symlink paths (the dev
checkout, the broken-symlink edge case, missing paths) fall through
unchanged.

### What the live install nevertheless exposed (P-034 + P-035)

Despite all of the above going green, the Mini's `guardian.log`
surfaced two new failure modes inside the post-Qwen knowledge-update
block that the test suite did not catch because they were live-only
shape-drift / path-drift problems:

- **P-034 — `core/mining_guardian.py:2409` hard-coded
  `/root/Mining-Guardian/knowledge.json`** in the Qwen post-scan
  persistence block. That path is a Linux/legacy dev path that does
  not exist anywhere in the Mac install tree (the install tree is
  `${MG_INSTALL_ROOT}` = `/Library/Application Support/MiningGuardian`).
  Every scan logged `[Errno 2] No such file or directory:
  '/root/Mining-Guardian/knowledge.json.tmp'` immediately after a
  successful Qwen call — meaning the Qwen response was generated but
  then **dropped before reaching `knowledge['llm_scan_analyses']`**
  every single scan. Vision Anchor 1 silent regression on top of an
  otherwise clean install. **Fix in P-034:** the block now resolves
  the path via `_ROOT / "knowledge.json"` and routes the
  read-modify-write through `core.file_lock.locked_knowledge_update`.
- **P-035 — `KnowledgeManager.update_from_scan` raised `KeyError:
  'total_flags'` on three of the 96 seeded `miner_profiles`** because
  the seed shape is not strictly uniform across all 96 entries
  (live evidence: miners `65891`, `65993`, `65958` lack `total_flags`).
  The line `profile["total_flags"] += 1` raised `KeyError`; the
  surrounding try/except in `core/mining_guardian.py:2443` swallowed
  it and logged `WARNING Knowledge update skipped: 'total_flags'` —
  silent skip of the chronic-miner accumulation for those three
  miners every scan. **Fix in P-035:** `update_from_scan` now
  `setdefault`s `total_flags` (0), `last_flagged` (None), and
  `issue_history` ([]) on the existing profile before incrementing.

hand-rolled `tmp + os.replace` writers of `knowledge.json` and one
non-atomic `KNOWLEDGE_PATH.write_text(...)` writer were all migrated

the Qwen scan analyzer, KnowledgeManager, daily deep dive, weekly
trainer, refinement chain, and outcome checker no longer race.

## Failure modes spotted this session

- **Build host vs agent bridge mismatch.** The agent bridge could
  not invoke `pkgbuild` against the build Mac today; the operator
  had to run the build, sign, notarize, staple, and spctl cycle
  from the Mac Terminal directly. This is consistent with prior
  weeks (the build path has always required Bobby's Mac); the
  notable bit is that the rest of the pkg build runbook needs to be
  written assuming "operator runs the commands; agent watches the
  log" — not "agent runs the commands." Captured here so future
  sessions don't try to delegate `pkgbuild` to the bridge.
- **Tailscale SSH port 22 to the Mini blocked from the build Mac.**
  `ssh miningguardian@100.69.66.32` returned `Operation not
  permitted` today, which prevented the catalog parity audit from
  running live. The Mini is reachable from the operator's machines
  and from inside the customer LAN; this looks like a session
  permission issue, not a Mini-side regression. Captured so that a
  future session reaching for `ssh ... 100.69.66.32` knows the
  fallback is "ask the operator to run the audit on the Mini
  directly."
- **Same P-NN reused for distinct work in the same row in
  MG_UNIFIED_TODO_LIST.md.** Row 10x (P-029 knowledge — installer
  payload) and rows 10u / 10v (P-029 ParameterRule and P-029
  shell-safe .env, both merged earlier in the week) all carry the
  P-029 prefix because the operator's "Round 9b" debugging spawned
  three distinct ad-hoc tickets that all kept the original
  P-NN. The unified TODO list explicitly flags this in row 10x's
  description ("NOT to be confused with row 10v / 10u; same
  P-number was reused for distinct work"). Lesson: when a P-NN is
  reused, the TODO row description has to disambiguate; cross-
  referencing branches by name (e.g.
  `mg/p029-knowledge-json-installer-payload`) is what makes the
  history readable.
- **`os.replace` preserves source-side mode bits.** P-032's root
  cause — `tempfile.mkstemp` creates files at 0600 and POSIX
  rename inherits source mode — is a well-known footgun, but the
  scanner had been writing the discovery sink at 0600 silently for
  weeks because nothing on the producer side reads it (the catalog
  importer reads it, and runs under a different account). The
  symptom only appeared once P-027's postinstall normalization
  healed the file to 0664 and the next scan rewrote it to 0600
  within the hour. Lesson: any atomic-write helper that needs a
  specific mode must `os.chmod` the temp file before `os.replace`,
  not after.
