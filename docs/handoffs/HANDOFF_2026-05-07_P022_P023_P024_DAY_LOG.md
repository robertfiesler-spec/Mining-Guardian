## P-022 — file-sink scanner discovery findings (PR #155)

### Why

The scanner emits "discovery" events when it sees a miner whose model
is not in the catalog or whose firmware version is unfamiliar. Pre-
P-022 those events only went to logs. Without an on-disk artifact:

1. The **daily catalog updater** could not consume them on its own
   cycle without parsing free-text logs.
2. The **operator** had no way to enumerate "what new things did the
   scanner see today?" except by tailing logs.
3. Restart of the daemon lost the in-memory list.

This violates Rob's standing rule "leave no data behind."

### What changed

Branch: `mg/p022-no-wasted-intelligence-data`. PR #155.

| File | Change |
|------|--------|
| `core/discovery_sink.py` | **New.** Atomic JSON snapshot writer + JSONL appender for discovery events. Configurable sink dir via `MG_DISCOVERY_SINK_DIR`, defaulting to `${INSTALL_ROOT}/cron_tracking/scanner_discovery/`. Writes `latest_findings.json` (atomic rename) plus `events-YYYY-MM-DD.jsonl` (append). |
| `core/mining_guardian.py` | Calls `record_discovery()` from the scan loop whenever the scanner produces an `unknown_model` or `new_firmware` event. |
| `tests/test_p022_discovery_sink.py` | 19 unit tests covering atomic-write semantics, sink-dir resolution, day-rollover, and macOS `/private/tmp` realpath quirk. |
| `docs/SCANNER_DISCOVERY_SINK.md` | New design + operator doc. |
| Daily catalog import wrapper | Adjusted to read the discovery/enrichment sweep path. |

Commit `7e111df` (feat) + `ce9831c` (test fix for macOS `/tmp` →
`/private/tmp` realpath portability — the test was passing on Linux

## P-023 — VPS runtime dependency audit + hardening (PR #156)

### Why

Rob asked: is the Hostinger VPS (`187.124.247.182`) anywhere in the
customer Mac Mini operating path? Specifically, is `daily_deep_dive`
making any VPS-side call (SSH, SCP, rsync, HTTP upload, callback
URL, etc.)?

The premise is that the cutover decision (D-9, "data plane stays
local") may have left orphan code that still gestures at the VPS.
Before removing Rob's PC from the operator path (P-024), the VPS
question had to be settled, because both sit on the same Tailscale
fabric and their fates were entangled in older designs.

### Audit method

Read-only sweep:
1. Grep for `187.124.247.182`, `100.110.87.1`, `fieslerfamily.com`,
   `tailscale`, `scp`, `rsync`, `ssh`, public hostname patterns,
   `/root/Mining-Guardian` — across the entire repo.
2. Trace every plist `ProgramArguments` and every launcher script in
   `installer/macos-pkg/resources/launchd/`.
3. Trace `daily_deep_dive` from entrypoint
   (`ai/daily_deep_dive.py:run_daily_deep_dive` line 940) through
   every external call: local Postgres (`GUARDIAN_PG_*`), local
   Ollama (`OLLAMA_URL` default `http://127.0.0.1:11434`), direct
   Anthropic API (`https://api.anthropic.com/v1/messages`), local
   `knowledge.json` write (`daily_deep_analyses` key).

### Audit conclusion

**Customer Mac Mini deployment has zero VPS runtime dependency.**

- No active code path in the daily deep dive (or anywhere in the
  scanner / API / console / catalog) calls the VPS.
- The only repo-resident reference inside the customer-shipped
  payload that named the VPS at runtime was
  `scripts/send_deep_dive_report.py` — a pre-cutover uploader with a
  hardcoded `2026-04-16` date and no caller anywhere in the active
  code path. Dead.
- `scripts/backup_db.sh` and `scripts/backup_mining_guardian.sh`
  reference the VPS but are operator-side tooling — they pull FROM
  the VPS TO Bobby's Mac. They were shipping in the customer payload
  via the broad `scripts/***` rsync filter, which P-024 then
  addressed.
- `fieslerfamily.com` is **operator-tunnel only** (D-19 console,
  Slack actions Cloudflare tunnel, dashboard CORS allowlist). Not on
  the customer data plane.
- The `_http_fallback_url` defensive guard in
  `ai/catalog_context.py:616-637` actively REFUSES any URL containing
  `100.110.87.1` (retired ROBS-PC tailscale endpoint).

### Audit conclusion

**Mini launchd / plists / launchers are clean** (every path is
`/Library/Application Support/MiningGuardian/...`, never
`/Users/BigBobby` and never a Tailscale IP). The Mini has zero
runtime dependency on Rob's PC or Bobby's Mac.

**The contamination was at the package boundary, not in the runtime.**
The customer payload was over-shipping via `--include 'scripts/***'`:
- `scripts/backup_db.sh` → references `BigBobby`, `100.103.185.53`,
  `187.124.247.182`, `/Volumes/Big-Bobby-T9/...`

- `scripts/branding/grafana_brand_dashboards.py` → operator dev tool
- `scripts/diagnostics/diagnose_ah3880_v2.py` → operator dev tool

None of these are referenced by any plist or launcher. None would
ever execute on the customer Mini. But they ride the .pkg payload,
visible if anyone unpacks it.
