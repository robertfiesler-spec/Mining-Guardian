## P-030 — early sys.path for `ai/` (PR #162)

### Why

The scanner's `run_once()` path is exercised both from the
postinstall baseline-scan step (`sudo -u miningguardian python
"${MG_INSTALL_ROOT}/core/mining_guardian.py" --once`) and from the
LaunchDaemon `com.miningguardian.scanner` plist. In both contexts,
the working directory is not the install root. `core/mining_guardian.py`
imports `from knowledge_manager import …` expecting the unpacked
sibling directory `ai/knowledge_manager.py` to be on the import
path; under both contexts that import resolved against the system
`sys.path` instead and silently fell into the no-op fallback that
logs `Knowledge update skipped: No module named 'knowledge_manager'`.

This is a Vision Anchor 1 issue: the scan loop is the entry point
into the LLM's learning loop. If `knowledge_manager` cannot be
imported, the scan executes but no knowledge update is produced,
and the local LLM does not get smarter from that scan.

### What changed

Branch: `mg/p030-ai-syspath-run-once`. PR #162.

| File | Change |
|------|--------|
| `core/mining_guardian.py` | Added an early `sys.path.insert(0, str(_ROOT / "ai"))` (after `_ROOT = Path(__file__).resolve().parent.parent`) so the bundled `ai/` package directory is searched before any other path. The insert runs at module import time, before `from knowledge_manager import …`. |
| `tests/test_ai_syspath_run_once.py` | New file with 5 assertions: (a) `ai/` is the first non-empty entry in `sys.path` after `core.mining_guardian` is imported, (b) `knowledge_manager` resolves to `ai/knowledge_manager.py` after the insert, (c) `run_once()` does not raise `ModuleNotFoundError` when invoked under a CWD that is not the install root, (d) the import order is stable across re-imports (idempotent insert — guards against duplicate entries on hot-reload), (e) the scanner still runs when `_ROOT / "ai"` does not exist (graceful degradation: the insert is best-effort, not a hard requirement). |

## P-032 — discovery sink atomic-write mode (PR #163)

### Why

P-027 installed `step_normalize_discovery_sink_perms` in the
postinstall to chmod `${MG_INSTALL_ROOT}/cron_tracking/scanner_discovery/latest_findings.json`
to 0664 on install/upgrade — so cross-account cron readers (the
catalog importer) could read the snapshot the scanner produces. But
on the post-P-028 Mini, the file was observed at 0600 within the
hour after install. Why?

`core/discovery_sink.py::_atomic_write_json` writes a temp file via
`tempfile.mkstemp(suffix=".tmp", dir=parent_dir)` and then calls
`os.replace(tmp_name, target)`. Two things matter:

1. `tempfile.mkstemp` creates the temp file with mode 0600 by
   design (security default).
2. POSIX `rename` (and Python's `os.replace`) **preserves the
   source file's mode bits when overwriting the destination**. So
   the destination `latest_findings.json` inherits the temp file's
   0600 every time the scanner writes.

The postinstall normalization healed the file once on install; the
next scan within the hour rewrote it back to 0600. The catalog
importer, which runs under a different macOS account, silently
skipped the 0600 snapshot.

### What changed

Branch: `mg/p032-discovery-sink-atomic-write-mode`. PR #163.

| File | Change |
|------|--------|
| `core/discovery_sink.py` | One-line addition — `os.chmod(tmp_name, 0o664)` between `os.fsync(tmp_fd)` and `os.replace(tmp_name, path)` in `_atomic_write_json`. The chmod is umask-independent (explicit numeric mode), so the behavior holds under both umask 022 (server default) and umask 002 (developer-typical). Doc-comment at the top of the function expanded to cite P-027 and explain why the explicit chmod is required despite the file being daemon-written. |
| `tests/test_p022_discovery_sink.py` | Extended with `TestAtomicWritePermissions` (4 new tests): (a) first write under umask 022 lands at 0664, (b) first write under umask 002 lands at 0664, (c) every subsequent scan keeps the file at 0664 (the actual bug — once-on-first-write would not have been enough), (d) a pre-existing 0600 file on disk (the broken pre-P-032 state customer Mini will have) is healed to 0664 by the next scan. All 4 use `os.umask` saving + restoring with `try`/`finally`. |

No `build_pkg.sh`, payload, postinstall, schema, migration, or
notarization changes — pure 1-line Python fix in one helper, plus a
comment block.

### Out of scope (deliberately narrow)

The `events-YYYY-MM-DD.jsonl` audit-trail file is created via
`open(..., "a")` (Python file open, no `mkstemp`), so its mode
depends on the daemon's umask, not on P-032's chmod path. That is
a separate (and much smaller) drift if it exists, and is NOT
addressed here per the narrow-scope instruction. P-027's postinstall
already chmods the JSONL on install/upgrade, so the cross-account
read works on day 0; if the umask drifts in a future session it
will need its own narrow fix.

### D-29 in detail (the contract that locks the work)

D-29 was added to `docs/DECISIONS.md` as part of this PR. The full
text is in the file; the highlights:

1. **Repo-side seed source path is locked** at
   `installer/macos-pkg/resources/knowledge/knowledge.json`. Future
   seed refreshes overwrite this file in-place; the build-time
   validation gate (step 4l) prevents a corrupt seed from reaching
   notarization.
2. **Payload staging path is locked** at
   `<payload>/installer-resources/knowledge/knowledge.json`.
3. **Runtime active path is locked** at
   `${MG_INSTALL_ROOT}/knowledge/knowledge.json`. A symlink at
   `${MG_INSTALL_ROOT}/knowledge.json` points to the active file so
   existing callers keep working without code change.
4. **Fresh install behavior** — postinstall creates the directory
   tree, copies the seed, sets ownership/mode, validates JSON,
   creates the symlink, emits the proof log line.
5. **Upgrade behavior** — postinstall does NOT overwrite. Site-
   specific learned `operator_decisions`, `refined_insights`,
   `baselines`, `llm_scan_analyses` survive. The packaged seed is
   staged alongside as `incoming/knowledge-seed-<version>-<sha>.json`.
6. **Monthly updates** are operator-staged (Tailscale `scp` or
   USB), not installer-driven. The merge implementation
   (`ai/apply_master_update.py`) is a follow-up ticket; the merge
   contract is locked here.
7. **Section-level merge rules** locked, so the implementation
   cannot drift:
   - `miner_profiles`, `miner_fingerprints`: dict-keyed, prefer
     newer `last_updated`. **Never delete site keys absent from
     master.**
   - `refined_insights`, `pattern_rules`: list, append-unique by
     `id`. Conflict on same `id`: prefer newer `last_updated`.
   - `operator_decisions`, `baselines`, `operator_rules`: **site-
     only** — master must not include them.
   - `llm_scan_analyses`: dict by scan timestamp. Append-unique.
   - `cross_miner_analysis`: list, append-unique by `id`. Index
     `[0]` is owned by the local Sunday weekly training and must
     not be replaced from master.
   - `daily_deep_analyses`: list by `date`. Append-unique. **Never
     deleted.**
   - `hardware_facts`: dict, top-level key overlay (master wins on
     shared keys; site-only keys preserved).
   - `last_updated`: ISO-8601, set to `max(site, master, now())`.
   - `fleet_summary`: recomputed from `miner_profiles` post-merge.
8. **Tombstone policy (v1):** the merge MUST NOT delete any local
   section key. A future `tombstones` field is reserved for a later
   PR with explicit operator review; v1 ignores it.

## P-033 investigation — recommendation: defer, do NOT change sink semantics

A separate investigation today looked at whether the discovery
sink's semantics needed to change to reduce the volume of
`unknown_model` and `new_firmware` events the scanner emits per
hour. The conclusion: **do NOT change sink semantics.** The events
JSONL stays append-only audit; `latest_findings.json` stays key-
deduped with an observation count.

The actual underlying issue is **producer-side**, not sink-side:

1. **AMS short codes are missing from the producer's known-model
   set.** The scanner's `_check_discoveries` compares miner-reported
   model strings (`S19JPro`, `S21EXPHyd`, `S21Imm`, `AH3880`)
   against a set populated from the catalog. The AMS short codes
   were never added to that set, so every scan emits an
   `unknown_model` event for every miner. The fix is to seed the
   short codes into the catalog (or load them from
   `intelligence-catalog/seed-data/aliases/003_live_short_name_aliases.sql`)
   and have the producer compare against the alias-resolved name.

2. **`core/database_pg.py::load_known_firmware` returns tuples while
   the producer compares against strings.** Pre-P-033 the load
   helper returned `[(version_string, model_id), …]` and the

## Lessons captured for future sessions

1. **`os.replace` preserves source mode bits.** When using
   `tempfile.mkstemp` + `os.replace` for atomic writes, the temp
   file is created at 0600 and the destination inherits it. If you
   need a specific mode, `os.chmod` the temp file BEFORE
   `os.replace`. Postinstall normalization alone is insufficient
   because the daemon will re-write the file within the next scan
   window.
2. **Postinstall + LaunchDaemon CWDs are NOT the install root.**
   Any Python module that imports a sibling package (e.g. `ai/`
   from `core/mining_guardian.py`) must add the install root's
   subdirectories to `sys.path` explicitly. The pre-P-030 import
   silently fell through to a no-op log line; the failure mode was
   "the LLM stops learning and nothing in the logs screams about
   it."
3. **A 32B fallback model the installer never pulls is a 100%
   failure rate, not a degraded mode.** Prefer the smallest D-13
   tier as the fallback so the failure mode is "older model than
   we'd pick" rather than "404 every scan."
4. **Filesystem contracts deserve a D-N entry the moment they're
   locked.** D-29 was added in the same PR as the postinstall step
   so the contract cannot drift. The section-level merge rules are
   load-bearing for the future federated implementation; if they
   were left to be "captured later," they would have drifted.
5. **`pkgbuild` runs on the build Mac, not via the agent bridge.**
   This is the same lesson from prior weeks; today reinforced it.
   When the pkg needs to be built, the runbook is "operator runs
   the commands; agent watches the log."

### P-036 PR opened, reviewed, merged the same evening

PR #169 (`mg/p036-knowledge-symlink-preserve-canonical-resolver`,
squash `906fa4a`, merge `fab6694`) changes ONLY `core/file_lock.py`.
A new private helper `_resolve_write_target(path)` returns the
symlink target (if the path is a symlink whose target's parent
directory exists) or the path itself otherwise.
`locked_knowledge_update` and `atomic_write_json` both route
through it: temp file lands in the **target's** parent and
`os.replace` lands on the canonical file, leaving the symlink
itself untouched. The `<path>.lock` lock file is keyed off the
resolved target so two writers — one given the symlink, one given
the canonical path — serialize against the same flock. Non-symlink
paths (dev checkout, broken symlinks, missing paths) fall through
unchanged.
