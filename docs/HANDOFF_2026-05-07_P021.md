**Status:** Ready to build — tests green, no live install attempted.

## What P-021 fixes

The first successful P-019E install on the Mini exposed three classes of catalog-DB drift the installer-acceptance phase masked:

1. **Catalog read paths used `WHERE model_id`** against tables whose canonical FK is `miner_model_id` / `primary_model_id` / `affected_model_id`. The scanner crashed on every catalog read with `column "model_id" does not exist` (B-30).
2. **AMS short names had no aliases** in `hardware.model_aliases`. The 4 short names from the BiXBiT USA fleet (`S19JPro`, `S21EXPHyd`, `S21Imm`, `AH3880`) silently failed Tier-1 lookup — the frozen-UUID Tier-1 seed dropped them on the FK gate (B-29) and they were never captured anywhere else (B-31).
3. **Daily catalog updates had no automated path.** The 5 Perplexity watchers wrote findings to `cron_tracking/<watcher>/`, but nothing on the Mini imported them into the catalog DB — `catalog_updater.py --add-from-csv` had to be invoked manually, which never happens on a customer install.

## Architectural anchor (from this batch's user clarification)

**Two DBs stay distinct, cooperating as one system.** Operational `mining_guardian` and catalog `mining_guardian_catalog` are NOT collapsed. They share the same Postgres container (`mining-guardian-db`), the same role (`mg`), and the same secrets (`MG_DB_PASSWORD`), but they have different schemas, different ownership boundaries, and different update cadences. P-021 makes them cooperate via clean contracts (schema-correct queries) + a daily import job, NOT via schema merge.

## What P-021 deliberately does NOT do

- **Does NOT regenerate the frozen-UUID Tier-1 seed** (B-29). That's the `uuid_generate_v5(NAMESPACE_OID, canonical_name)` rewrite of `seed_miner_models.sql` + alias-seed regenerator, which is multi-PR work and risks customer-data re-keying. Out of P-021 scope.
- **Does NOT migrate `unified_miner_index.json` / `miner_enrichment_master.csv` into Postgres**. Per D-12 + D-20, Postgres is the source of truth and those JSON/CSV files are operator-side debug exports. The `catalog_updater.py --add-from-csv` path lights up DB writes; the file artifacts are the audit trail, not a parallel source.
- **Does NOT rebuild the .pkg.** The user explicitly asked: prepare the branch, validate, do not build/sign/notarize/transfer/install.
- **Does NOT touch the four other watchers' DB writers** (Aggregator, Manufacturer Model, Firmware Tracker, Community Intel). Those remain JSON/CSV-only — see `docs/INTEL_CATALOG_FULL_BRIEF_2026-05-02.md` §351-354 for why each is queued for a future PR.
- **Does NOT fix the Qwen 404 / `knowledge_manager` warnings** observed in the P-020 install logs. Those are scanner runtime warnings unrelated to the catalog DB; they belong in a future P-022 scanner-warning sweep.

## Remaining risks

1. **Tier-2 fallback is still the production resolver.** The supplement only adds 4 BiXBiT-specific short names. Other customer fleets emitting different AMS short forms will need entries appended to `003_live_short_name_aliases.sql`, OR Tier-2 will catch them.
2. **`run_daily_catalog_import.sh` runs as a launchd LaunchDaemon (root)**, but `catalog_updater.py` writes to `intelligence-catalog/data/unified_miner_index.json`. Need to verify file ownership of `intelligence-catalog/data/` permits root write (postinstall's `step_install_launcher_wrappers` chowns `bin/` root:wheel; `intelligence-catalog/` ownership is install-default). Worst case: the daily run fails on first execution, the operator sees the ERROR in `logs/scheduled/catalog_import.err.log`, and we adjust postinstall in P-022.
3. **Tier-1 supplement file failure is non-fatal** by design (logged ERROR, install continues). If a customer Mini somehow gets a half-applied supplement, the next reinstall will heal it via `ON CONFLICT DO NOTHING`.
4. **Future contributor footgun**: any new catalog FK column must be named `miner_model_id` to match the de-facto convention. The schema-contract test catches the regression class but not at design time.
