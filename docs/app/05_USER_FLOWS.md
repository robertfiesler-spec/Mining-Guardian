# 05 — User Flows (v1)

The five flows the v1 app must support. Each one is described from the operator's perspective, with API hooks listed so we know exactly what we're plumbing.

**Reading shorthand:**
- `→ /api/...` means "the app calls this endpoint"
- `[screen: X]` means "user lands on screen X"
- "the AI" means the local Ollama (`qwen2.5:7b`) — never a cloud LLM

---

## Flow 1 — Approve / Deny an alert (the highest-value flow)

**Trigger:** A miner overheats. The scanner daemon catches it, the AI proposes "throttle to 80% for 30 min", and the alert lands in two places: a Slack DM (today's behavior, kept) and the new app's inbox.

### Steps

1. **Operator gets Slack ping**, half-asleep at 2am.
2. Operator opens the app on their phone (already logged in via "remember device").
3. **`[screen: Inbox]`** shows: 1 unread alert, top of list, status pill **● WARN**.
4. Operator taps the alert.
5. **`[screen: Alert detail]`** shows:
   - Miner ID (e.g. `S21-Pro-014`), location ("Rack B-3"), age ("8 minutes ago")
   - Symptom: "Hashboard 2 temp 87°C, 12°C above 30-day median"
   - Proposed action: **Throttle to 80% for 30 min** (orange call-out card)
   - AI's reasoning: 2-sentence explanation generated by local Ollama
   - Confidence indicator: "AI confidence: high (similar pattern resolved 23 times)"
   - Three buttons: **APPROVE** (orange), **DENY** (silver outline), **CUSTOMIZE** (silver outline)
6. **Path A — APPROVE:**
   1. Operator taps APPROVE
   2. Optional: free-text field "anything to note?" (skippable, autofocus = no)
   3. Confirm sheet: "Throttle S21-Pro-014 to 80% for 30 min — confirm" with red Cancel and orange Confirm
   4. App calls `POST /api/approvals/{id}/approve` → existing `approval_api.py:8686`
   5. Toast: "Action queued. Throttle in progress."
   6. Status pill on the alert flips to **● APPLYING** (info blue, pulsing)
   7. ~10 seconds later, daemon reports back via SSE: pill becomes **● RESOLVED** (success green)
   8. Operator closes phone and goes back to sleep.
7. **Path B — DENY:**
   1. Operator taps DENY
   2. Sheet appears: 4 canned reasons (radio) + free-text fallback
      - "Not the issue — investigating elsewhere"
      - "Fixing manually right now"
      - "Miner is in maintenance — ignore alerts"
      - "False positive"
   3. Operator picks one, optionally adds note, taps **DENY** in sheet
   4. App calls `POST /api/approvals/{id}/deny` with reason payload
   5. Status pill flips to **● DISMISSED** (muted grey)
   6. The deny reason is logged to `intelligence_catalog.feedback` for the AI to learn from
8. **Path C — CUSTOMIZE:**
   1. Operator taps CUSTOMIZE
   2. **`[screen: Customize action]`** lets operator change the throttle target (60–95%) and duration (5–120 min) with sliders
   3. Operator hits APPROVE — same flow as path A but with custom params

### API endpoints needed

- `GET /api/inbox` (list of pending + recent alerts) — new
- `GET /api/alerts/{id}` — new (or proxies existing approval_api endpoint)
- `POST /api/approvals/{id}/approve` — proxies `:8686/approve`
- `POST /api/approvals/{id}/deny` — proxies `:8686/deny`
- `GET /api/sse` — SSE stream for status updates

### Edge cases (build for these)

- Operator approves twice (network retry) → server is idempotent, second call returns 200 with same result
- Slack approval beats app approval → app sees status update via SSE, button becomes "Approved in Slack — view"
- Daemon offline → app shows "Action queued; will apply when scanner reconnects" instead of failing
- Alert auto-resolves before operator opens it (miner cooled down) → status is **● AUTO-RESOLVED** (info), no buttons

## Flow 2 — Browse the fleet

**Trigger:** Operator wants to see "what's running right now."

### Steps

1. Operator taps **Fleet** in nav.
2. **`[screen: Fleet]`** shows:
   - Summary tiles at top: **Total: 247** | **Alive: 241** | **Warn: 4** | **Offline: 2**
   - Aggregate hashrate: **24.7 PH/s** (orange) | Power: **18.4 kW** (silver) | Site temp: **24°C**
   - **"Scan now"** button (orange, top-right)
   - Filter chips: All / Warn / Offline / Stale / By rack
   - Sortable table: ID | Status | Hashrate | Temp | Power | Last seen | Actions
3. Operator taps a row → **`[screen: Miner detail]`**
4. Miner detail shows:
   - Big status pill at top, miner ID, hostname, IP
   - 24-hour hashrate sparkline
   - Temperature triplet (3 hashboards) with color-coded bars
   - Last 5 alerts for this miner
   - Action buttons: **Restart**, **Throttle...**, **Mark in maintenance**, **View in Grafana** (deep link)
   - Footer: AMS link, firmware version, location

### API endpoints needed

- `GET /api/fleet` (list, sortable/filterable via query params) — proxy or extend existing `dashboard-api`
- `GET /api/fleet/{id}` (single miner with 24h history)
- `POST /api/fleet/{id}/scan` (trigger immediate single-miner scan)
- `POST /api/fleet/scan-all` (trigger full fleet scan)

### Edge cases

- Empty fleet (fresh install) → empty state with shield-silver illustration and copy "No miners yet. Add credentials in Settings → AMS."
- 500+ miners → virtualize the table (react-window) so scroll stays at 60fps
- Stale data (last scan >5 min ago) → banner at top: "Showing data from N minutes ago. Scan now."

## Flow 3 — Set or change the schedule

**Trigger:** Operator wants the scanner to run every 5 min normally but every 1 min during peak heat hours.

### Steps

1. Operator taps **Settings → Schedule**.
2. **`[screen: Schedule]`** shows current schedule as a 24h timeline:
   - Visual: 24-hour horizontal bar, segmented by current cadence rules
   - List of rules below: "Mon–Fri 09:00–18:00: every 5 min", "Daily 12:00–16:00 (peak): every 1 min", "Daily 22:00–06:00: every 30 min"
3. Operator taps **+ Add rule**.
4. Rule editor (modal):
   - Days: Mon-Sun multi-select
   - Time range: start / end (24h pickers)
   - Cadence: dropdown (1, 5, 10, 15, 30, 60 min)
   - Optional: "Pause scans" toggle (for blackout windows like maintenance)
   - Save / Cancel
5. Operator saves → app calls `PUT /api/schedule/rules` with full rule set
6. Backend writes to `system_schedules` table (migration 005), reloads scanner daemon's cron via `launchctl kickstart -k`
7. Toast: "Schedule updated. Next scan in 4 min."

### API endpoints needed

- `GET /api/schedule` (current rules)
- `PUT /api/schedule/rules` (replace rule set; transactional)
- `POST /api/schedule/preview` (dry-run: "given these rules, when will the next 10 scans run?")

### Edge cases

- Overlapping rules → server resolves with "most specific wins; ties go to first declared"; app shows a warning if it detects ambiguity
- Operator deletes all rules → server falls back to default (every 15 min); app shows "Default schedule restored"
- Pause window currently active → fleet view shows banner "Scanner paused until 03:00"

## Flow 4 — Read the Intelligence Report

**Trigger:** Operator wants to know "did we make money today?"

### Steps

1. Operator taps **Reports** in nav.
2. **`[screen: Intelligence Report]`** shows the latest report (regenerated daily by the `intelligence-report` daemon at 06:00 local):
   - **Hero number:** "**$ 184.32 net** today" (orange if positive, red if negative)
   - Sub-line: "247 miners online · 24.7 PH/s · 18.4 kW · BTC $96,430"
   - **AI summary card** (3-paragraph human-readable summary from local Ollama)
   - **Profitability breakdown:** revenue / power cost / net (small bar chart)
   - **Health snapshot:** uptime %, alerts triggered, alerts auto-resolved, alerts requiring approval
   - **Recommendations card:** "AI suggests: 2 miners on Rack B running below 92% — consider firmware refresh"
   - **Time selector:** Today / Yesterday / Last 7 days / Last 30 days
   - **Download PDF** button (top-right)
3. PDF download → `GET /api/reports/{id}/pdf` returns a brand-styled PDF for sharing

### API endpoints needed

- `GET /api/reports/latest`
- `GET /api/reports?range=7d`
- `GET /api/reports/{id}/pdf` — runs the PDF builder (existing? new? — check)

### Edge cases

- Report not yet generated for today (before 06:00) → show yesterday's with banner "Today's report at 06:00"
- BTC price feed fails → use last cached price, footnote "BTC price as of [time], cached"
- AI summary fails → fall back to template-based summary (no LLM), tagged "Generated from data only"

## Flow 5 — Change a setting

**Trigger:** Operator rotates Slack token after a security review.

### Steps

1. Operator taps **Settings**.
2. **`[screen: Settings]`** shows tabs: **General** | **AMS** | **Slack** | **Alerts** | **Operator Mode** | **Backups** | **About**
3. Operator taps **Slack** tab:
   - Bot token (masked, "reveal" button shows once)
   - Channel for alerts (dropdown populated from Slack workspace)
   - Channel for approvals
   - "Test connection" button → posts a test message and reports success/fail
4. Operator updates token, taps Save.
5. App calls `PUT /api/settings/slack` with the new values; server validates by hitting Slack's `auth.test`, then writes to encrypted `app.settings` rows.
6. Toast: "Slack settings saved. Test message sent."
7. Audit log entry recorded: "Operator updated Slack settings at [timestamp]"

### API endpoints needed

- `GET /api/settings` (all settings, secrets masked)
- `PUT /api/settings/{namespace}` (per-tab save)
- `POST /api/settings/slack/test` (test connection)
- `POST /api/settings/ams/test` (test AMS credentials)

### Operator Mode tab (special)

- Big toggle: **Full Auto** (orange) | **Semi-Auto** (silver) | **Manual** (silver)
- Description for each mode
- Effect on the rest of the app: Manual mode hides "auto-resolved" alerts and shows everything as needing approval
- Saving this setting writes to `system_settings` table (migration 005), broadcasts via SSE so any other open tab updates immediately

### Edge cases

- Operator saves an obviously bad token → server validates with Slack first, returns 400 with "Slack rejected this token. Check that it starts with `xoxb-`."
- Operator changes mode mid-incident → app shows confirmation modal "There are 3 pending approvals. Switching to Manual will require you to handle each one. Continue?"
- Network failure during save → app keeps the unsaved changes locally, shows "Save failed — retry?" banner, retries on next online

- Custom alert rule builder (use defaults; advanced is post-v1)

## Cross-cutting flows

### Login (every session)

1. **`[screen: Login]`** — full-bleed shield logo, password field, "Remember this device" checkbox, sign-in button (orange).
2. Operator types password, hits Enter.
3. App calls `POST /api/auth/login` with `{password, device_name}`.
4. Server bcrypt-compares, on success returns httpOnly session cookie (24h or 30d depending on checkbox) and operator profile.
5. Redirect to last-visited screen, or **Inbox** if first login.

### Logout

1. Operator taps avatar → Logout.
2. App calls `POST /api/auth/logout` (deletes session row).
3. Cookie cleared.
4. Redirect to login.

### Error boundaries

- All API failures show a toast in the bottom-right corner with the error message and a **Retry** link
- Network offline → app shows top banner "Offline — last data X min ago"
- Daemon offline (we can detect via heartbeat endpoint) → top banner "Scanner offline — alerts paused"
