# Documentation Reconciliation Audit — 2026-05-16

> **This is a findings report, not a cleanup. Nothing was deleted to produce it.**
> It establishes a single source of truth per domain, catalogues evidence-backed
> drift between "canonical" docs and live ground truth (Mini / VPS / installer),
> and proposes the discipline that prevents recurrence. Acting on it = reviewed
> PRs afterward, deliberately, never a fast sweep.

## Why this exists

Two of the most damaging findings in the 2026-05-16 session — W35 (the daily
deep-dive launchd job absent on the Mini since cutover) and W36 (the
log-collection/deep-dive schedule shipped 3 weeks stale) — share one root
cause: **operational reality changed and the canonical documentation did not,
and nothing caught the drift.** The schedule change lived on the VPS crontab
for ~3 weeks while `docs/CRON_SCHEDULE.md` still said the old times; the
program ran wrong for weeks and no process flagged it. Separately, the B-6
typo guard was red on `main` for 2 days (since PR #223) because an allow-list
silently drifted from the docs it guards.

The disease is not "too many files." It is **no declared owner + no freshness
contract + no reconciliation step** for documents that nonetheless call
themselves canonical. This audit names the disease precisely and prescribes
the cure.

## 2 · Single-source-of-truth map (proposed)

For each domain, exactly ONE document is authoritative. Every other document
touching that domain must either (a) defer to the owner by reference, or (b)
be explicitly marked historical. This table is the proposed contract; adopting
it is a CLAUDE.md change (see §4).

| Domain | Proposed single source of truth | Demote / mark-historical |
|--------|----------------------------------|--------------------------|
| Daily/weekly **schedule** | `docs/CRON_SCHEDULE.md` (post-#228) — and it must be reconciled to the VPS crontab until the VPS is decommissioned, then to the Mini launchd plists | DECISIONS.md schedule lines → reference CRON_SCHEDULE.md; remove inline times |
| **Execution-plan / W-item status** | `docs/EXECUTION_PLAN_STATUS.md` | All handoffs/briefings reference it, never restate status |
| **Locked decisions** | `docs/DECISIONS.md` | — (but it must itself be reconciled; see D2/D3) |
| **Working conventions** | `CLAUDE.md` | — |
| **Deployment procedure** | **NEEDS DECISION.** `MAC_MINI_DEPLOYMENT_RUNBOOK.md` is stale (D4). Candidate: the installer (`postinstall.sh`) is now the *de facto* truth; the runbook should either be rewritten to the W14 topology or explicitly demoted to "historical, superseded by the .pkg installer" | Until rewritten, the runbook must carry a loud "DO NOT USE FOR W14+ INSTALLS" banner |
| **Live system state** | `docs/STATE_OF_THE_SYSTEM_*` (dated snapshots — inherently historical; newest is most current but none is "live") | — |
| **Latent bugs / guards** | `docs/LATENT_BUGS.md` (+ its guard allow-lists kept in lockstep — the B-6 lesson) | — |

## 3 · Highest-risk landmine — D4, called out separately

`MAC_MINI_DEPLOYMENT_RUNBOOK.md` is the one finding here that can cause a
fresh production failure if left alone, because its "canonical" self-label
actively invites someone to follow it. Recommended handling (its own tracked
item, operator-present, NOT folded into the audit PR — Failure Mode 9):

1. Decide: rewrite to W14 two-DB topology, or retire + point to the installer.
2. Whichever — until it is done, the runbook gets a top-of-file banner:
   `⚠️ STALE — predates W14 two-DB split (2026-05-13). Do NOT use for installs.
   The .pkg installer (installer/macos-pkg/) is the actual deployment path.`
3. That banner is a 1-line safety change that can ship immediately, separate
   from the larger rewrite/retire decision.

## 4 · The cure — proposed CLAUDE.md discipline

The root cause (D5) is the absence of a reconciliation contract. Proposed
addition to `CLAUDE.md` (its own PR, mirrors the access-kickoff #226 pattern
that successfully closed a similar process gap):

> **Single-Source-of-Truth discipline (added 2026-05-16, from DOC_AUDIT).**
> Each domain has exactly one authoritative document (see
> `docs/DOC_AUDIT_2026-05-16.md` §2). Two binding rules:
> 1. **Same-commit reconciliation.** A change to operational reality
>    (schedule, topology, job set, paths) is NOT done until the authoritative
>    doc for that domain is updated in the SAME commit/PR. This is the
>    document-as-you-go rule, sharpened: it names *which* doc.
> 2. **Ground-truth beats prose.** When a doc and the live system disagree,
>    the live system wins and the doc is the bug. Verify against ground truth
>    (Mini/VPS) before trusting a "canonical" label — the
>    `MAC_MINI_DEPLOYMENT_RUNBOOK` self-declared canonical while 2 weeks stale.

This does not require deleting anything. It makes drift a commit-time
obligation instead of a thing discovered weeks later by accident.

## 6 · What this audit deliberately did NOT do

- Did not delete any document, on any machine (laptop/Mini/VPS).
- Did not bulk-"clean" stale files — staleness was *catalogued*, not removed,
  because (proven this session) old/redundant content is sometimes the only
  ground truth (the VPS crontab is what resolved W36; a fast cleanup would
  have destroyed exactly that).
- Did not audit all 269 non-archive markdown files — it risk-prioritised the
  self-declared-canonical docs, where the damaging drift lives. A fuller pass
  is a future item if desired, but the high-value landmines are in §1.
